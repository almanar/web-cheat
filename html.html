    Standards

    HTML is standardized by the Word Wide Web Consortium (W3C), homepage http://www.w3.com

    The w3schools website is not affiliated in any way to W3C. It is said that it contains many errors, and its certification is worthless: it is the cplusplus.com of the web.

    W3C has many important member organizations including companies, universities and others such as Microsoft, Apple, IBM, Google, Facebook, Twitter, Adobe.

    W3C specifies many of the key web standards including:

    • HTML
    • CSS
    • DOM
    • XML
    • CGI

    One important standard that they do not specify is Javascript.

    W3C have several levels of endorsement. - WD (Working Draf) is the lowest level of endorsement of material available to the public - REC (Recommendation) is the highest level, for stable and widely accepted features. - NOTES are possible to make small later revisions. - retired is for standards that have been deprecated

    #HTML5

    WD of 2008, expected REC for 2014: HTML5 standard The huge time lapse is in part due to wars on the future of Flash replacement.

    Not all HTML5 is not XML, just pretty close to it. It is however possible to write HTML5 code that is valid XML. XHTML5 is a version of HTML5 which is always XML.

    HTML5 is not SGML, but it was influenced by it. HTML 4 is SGML.

    #XHTML5

    XHTML5 is a strictly XML version of HTML5.

    The standard is developed together with HTML5.

    Browsers differentiate between the two based on the HTTP content type: text/html is for HTML5 and application/xhtml+xml is for XHTML5. The DOCTYPE is not used to differentiate between HTML and XHTML.

    XHTML5 vs HTML5

    #End tags #Closing tags

    In XHTML5 all tags can be self-closing (e.g. <br>). All tags must be closed (self or not). All tags can be either self-closing or not.

    In HTML5 things are much messier:

    • Void elements cannot have a separate end tag, but they may have a meaningless slash / added to them (spec) so the following are all valid: <br>, <br/> and <br />.
    • Foreign elements can be self closing.
    • The other elements cannot be self closing.

    #entities #char references

      There are two kinds of references:

  - named. E.g.:   &gt;   >
  - numeric: Refer to Unicode code points. Start with the number sign `#`.
      - decimal:   &#97;  a
      - hex:       &#x61; a

  HTML5 adopts exactly the same references as XML, but earlier versions do not, so watch out.

  Entities are considered in both data and attribute values.

  Escape sequences intended for characters that:

  - have an specific meaning to HTML. Important ones:

      - greater than &gt;  >  Used in tags.
      - less than    &ls;  &ls;  Used in tags.
      - ampersand    &amp; & Used in entities.

  - not present in the current encoding, such as Unicode character in an ASCII encoded file.

      Prefer Unicode for all non-whitespace characters.

      Useful whitespace characters:

      - non-breaking whitespace #nbsp;   Whitespace at which lines cannot reflow.

    #id

    Before HTML5, id attributes must start with `a-zA-Z`. In particular, no digits. This restriction was lifted with HTML5, but it is still recommended to follow it for backwards compatibility.

    `id` attributes must be unique across the entire document: http://programmers.stackexchange.com/questions/127178/two-html-elements-with-same-id-attribute-how-bad-is-it-really

    Multiple classes are space separated as in: "class0 class1 class2"

    alt

    If:

    • an image does not exist
    • the format is not supported by the browser (support list)

    then alt the browser can show the alt text instead of the image.

    As of 2013, chrome replaces a missing image with a dummy image, therefore ignoring alt. Therefore don't rely on alt being shown in your app logic: alt should only be shown in case of error. If an image should not present, replace it with something else explicitly.

    According to the HTML5 validator, alt is mandatory except under certain conditions, so just use it always for your sanity.

    #frame #iframe

        Include another HTML document in the current one.
    This element therefore automatically makes an HTTP request to another page,
    so it is an interesting way go include HTML content without any Javascript.
    In the past, before server side scripting, widely used to reuse code fragments such as indexes.
    Now generally frowned upon for navigation.
    - `frame` was invalidated in HTML5.
    - `iframe`, inline frame, is still valid in HTML5.

    iframe always has a fixed size, which you can change with CSS:

    disable scroll in HTML5 CSS3 is not yet widely supported: http://stackoverflow.com/questions/1691873/safari-chrome-webkit-cannot-hide-iframe-vertical-scrollbar

    The frame content is like `alt` for images. Your browser may simply ignore it in case of failure if he has something better to do (Firefox 27 shows the not found page with a reload button.)

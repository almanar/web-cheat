    #HTML5

    WD of 2008, expected REC for 2014: HTML5 standard The huge time lapse is in part due to wars on the future of Flash replacement.

    Not all HTML5 is not XML, just pretty close to it. It is however possible to write HTML5 code that is valid XML. XHTML5 is a version of HTML5 which is always XML.

    HTML5 is not SGML, but it was influenced by it. HTML 4 is SGML.

    #XHTML5

    XHTML5 is a strictly XML version of HTML5.

    The standard is developed together with HTML5.

    Browsers differentiate between the two based on the HTTP content type: text/html is for HTML5 and application/xhtml+xml is for XHTML5. The DOCTYPE is not used to differentiate between HTML and XHTML.

    XHTML5 vs HTML5

    The distinction is made by the MIME type on the `content-type` field of the HTTP header:

    • XHTML uses `application/xhtml+xml`
    • HTML uses `text/html`

    #element vs #tag #start tag #end tag

    `<p>a</p>` is an element

    `<p>` and `</p>` are a start and end tag respectively. The standard does not mention the terms "opening" or "closing" tags, so don't use them.

    #Tag case

    All tag names are case insensitive.

    Lower case names are much more common now. They are easier to type, and tags + syntax highlighting already gives them enough visibility.

    XHTML5 is simple, just like XML:

    • all tags can be self-closed, e.g: <br>
    • all tags must be closed somehow (self or not).

    In HTML5, things are much messier. There are five different kinds of elements to consider, and tons of rules.

    • #Void elements: area, base, br, col, embed, hr, img, input, keygen, link, meta, param, source, track, wbr.
    • #Raw text elements: script, style
    • #Escapable raw text: textarea, title
    • #Foreign elements: MathML and SVG.
    • #Normal: the rest, including p, u, li, div

    #void element closing tag

    void elements are the only elements which *cannot* have an end tag.

    As a consequence, they cannot have any content, which justifies their name.

    #self closing slash

    • HTML5: only void and foreign elements may have a self closing slash.
      • void elements: `<br />` == `<br>`: the slash is omitted
      • foreign elements: same as XHTML
    • In XHTML, `<p />` == `<p /></p>`
    • HTML4: `<p />` == `<p>>`. The slash can appear on any element, and it is the same as the greater than sign.

    #optional tags #omit tags

    • start and end tags for some elements can be omitted: http://www.w3.org/TR/html5/syntax.html#syntax-tag-omission There is no simple rule, only a huge list of rules. Some interesting points include:
      • html, body and head *start and end* tags can be omitted!
      • p elements has by far the most complex rule. Quoting: p element's end tag may be omitted if the p element is immediately followed by an address, article, aside, blockquote, div, dl, fieldset, footer, form, h1, h2, h3, h4, h5, h6, header, hgroup, hr, main, nav, ol, p, pre, section, table, or ul, element, or if there is no more content in the parent element and the parent element is not an a element.
      • li element's end tag may be omitted if the li element is immediately followed by another li element or if there is no more content in the parent element.
      • div, ul, table and blockquote tags must have and end tag
    • Foreign elements can be self closing, but the meaningless slash is not meaningless.

    #protocol relative URL #URL without protocol

    E.g.: `//google.com` instead of `http://google.com`.

    Browser uses the same protocol as the current one.

    Useful to be dry when using HTTPS vs HTTP for resources that can be retrieved in both forms.

    CDNs often offer CSS and Javascript in protocol relative URLs.

    Specified in RFC 3986, with the rest of the URI specs: http://tools.ietf.org/html/rfc3986#section-4.2

    Drawback: if you open your files with the `file://` protocol, it will also use the `file` protocol and fail. Workaround: use a web server on `http://localhost`.

    Well implemented on modern browsers.

    #References #Character references #Entities

    HTML5 uses only the term "references": http://www.w3.org/TR/html5/infrastructure.html#syntax-references

    There are two kinds of entities:

    • named. E.g.: &gt; >
    • numeric: Refer to Unicode code points. Start with the number sign `#`.
      • decimal: &#60; <
      • hex: &#x3C; <
      • TODO: possible to use Unicode can be achieved directly like &#4E2D;?

    HTML5 adopts exactly the same references as XML, but earlier versions do not, so watch out.

    Notably: " is on the official list of valid HTML 4 entities, but ' is not.

    Entities are considered in both data and attribute values.

    Escape sequences intended for characters that:

    • Have an specific meaning to HTML. Important ones:
      • greater than &gt; > Used in tags.
      • less than &lt; < Used in tags.
      • ampersand &amp; & Used in escape entities themselves.

    • Not present in the current encoding, such as Unicode character in an ASCII encoded file.

      Prefer Unicode for all non-whitespace characters.

      Useful whitespace characters:

      • non-breaking whitespace #nbsp;   Whitespace at which lines cannot reflow.

    #IDL attribute vs #content attribute

    Not content attributes that are written in HTML tags, but rather internal properties that are accessible through Javascript properties of the DOM model. For this reason they are often called properties.

    Important confusing case: checked attribute vs property. Changing the attribute also changes the property, but clicking on the checkbox only changes the property, not the attribute.

    #categories #flow content #phrasing #which element can contain which

    HTML5 has many element categories.

    Flow content and phrasing are two of the most important categories.

    Elements can be both flow content and phrasing at the same time.

    One of the most important thing this determines is which elements an element can contain. Each element has:

    • a list of categories it belongs to
    • #Content model: a list of elements or categories it can contain

    Important conclusions that can be taken from content models and categories include:

    • html can only contain head followed by a body (but omission is possible for head and body)
    • head contains metadata category elements: base, link, meta, noscript, script, style, template, title
    • body contains flow content
    • p is flow content and can only contain phrasing elements, therefore it cannot contain: p, div, ul
    • ul can only contain li or script supporting elements
    • li contains flow content. It can therefore contain and `p`, `ul`.
    • div is and can contain flow content elements

    #block-level #inline #inline-block #replaced

    HTML4 uses terms block-level, inline and inline-block. In most senses, all elements are either block-level or inline.

    HTML5 has many more categories.

    • block-level corresponds roughly to HTML5 flow content
    • inline corresponds roughly to phrasing

    Inline replaced elements such as images which behave like elements with CSS `display:inline-block` property. They can be put inside other inline elements.

    #forms

    #form

    Contains everything needed to send data to the server, including inputs, input labels and a submit button.

    Important attributes:

    • `action`: where to send the request to
    • `method`: what HTTP method to use. Can only be:
      • get: send data on the URL URL encoded, which has size limitations.
      • post: sends data on the body URL encoded.
    • `enctype`: value of the `content-type` header
      • `application/x-www-form-urlencoded`: default. URL encode.
      • `multipart/form-data`: can only be used if the `method` is `post`. Best option for uploading large data such as files via `input type="file"`.
      • `text/plain`: useless?

    To see what forms do exactly, consider using `nc` or an ECHO server.

    Multipart form to http://example.com:

    Sending the form is like clicking a link: the browser attempts to render the reply, sets the URL on the address bar, etc. For this reason, the most common action after submitting a form is to redirect to somewhere else.

    #name attribute

    For most form elements, gives the name of the parameter that will be sent on th request.

    If missing:

    • the HTML is still valid
    • the value is not submitted

    #disabled attribute

    Any form field can have the disabled boolean attribute. If present, it:

    • prevents the value of the field from being changed
    • does not send the disabled field on form submit
    • if on submit button, prevents form from being submitted

    The disabled IDL attribute (Javascript property) always matches the content attribute.

    Even so, it is better to modify the IDL attribute instead of the content attribute, making things more uniform with checkbox operation, and avoiding unnecessary HTML modification.

    #boolean attribute

    Examples: disabled, checkbox's checked, select's selected.

    If not present: false. If present, true.

    Do not need a value, but if a value is given there are only two valid possibilities:

    • empty string
    • equal to the attribute case insensitive

    The best option is using an exact lowercase match of the attribute, which is also XHTML5 portable.

    Mostly historical elements: new elements like contenteditable get the saner true and false valid values.

    #readonly attribute

    Boolean attribute.

    If true, field cannot be modified, but unlike disabled will be sent with the form and its text can be selected.

    Invalid for checkboxes: http://stackoverflow.com/questions/155291/can-html-checkboxes-be-set-to-readonly Just disable them.

    #checked

    http://www.w3.org/TR/html5/forms.html#attr-input-checked

    Boolean attribute that determines the checkbox's initial checked state.

    If modified, also modifies the checked state if the checkbox hasn't been clicked on. It it has, a dirty flag is set, and adding or removing the attribute does not modify the checked state.

    Resetting a form resets the dirty flag.

    The conclusion is simple: only use this attribute to set the initial value of the checkbox: never modify it with Javascript. Use the checked property instead.

    Contrast this complex behavior with that of the disabled attribute, which always matches the IDL attribute.

    Beware! Some browsers like Firefox might remember the checkbox value from old submits, and override this attribute. Your logic should not depend on it's initial state. Ways to disable this: http://stackoverflow.com/questions/299811/why-does-the-checkbox-stay-checked-when-reloading-the-page

    #frame #iframe

    Include another HTML document in the current one.

    This element therefore automatically makes an HTTP request to another page, so it is an interesting way go include HTML content without any Javascript.

    In the past, before server side scripting, widely used to reuse code fragments such as indexes.

    Now generally frowned upon for navigation.

    • `frame` was invalidated in HTML5.
    • `iframe`, inline frame, is still valid in HTML5.

    iframe always has a fixed size, which you can change with CSS:

    Disable scroll in HTML5 CSS3 is not yet widely supported: http://stackoverflow.com/questions/1691873/safari-chrome-webkit-cannot-hide-iframe-vertical-scrollbar

    The frame content is like `alt` for images. Your browser may simply ignore it in case of failure if he has something better to do (Firefox 27 shows the not found page with a reload button.)
